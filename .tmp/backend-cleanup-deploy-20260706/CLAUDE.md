# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# Run migrations
python manage.py migrate

# Seed chart data from the master workbook (--clear wipes existing chart records first)
python manage.py seed_data --clear
python manage.py seed_data --workbook path/to/custom.xlsx

# Seed CMS configuration defaults (platforms, countries, roles)
python manage.py seed_cms_defaults

# Backfill artist country/country_code from the built-in metadata map
python manage.py update_artist_countries

# Run development server (SQLite, DEBUG=True by default)
python manage.py runserver

# Run all tests
python manage.py test charts

# Run a single test class or method
python manage.py test charts.tests.PublicAppDataSyncTests
python manage.py test charts.tests.PublicAppDataSyncTests.test_all_public_facing_cms_saves_feed_the_app_payload

# Finalize a completed month: appends it to the master workbook + updates MONTHS
python manage.py finalize_month --year 2026 --month 6 --dry-run   # preview first
python manage.py finalize_month --year 2026 --month 6              # then apply

# Export static chartData.js for the frontend
# Automatically applies Data/ngoma_duplicate_releases_final_merge_ready.xlsx if present
python scripts/export_frontend_data.py charts/seed_data/Ngoma_Charts_MASTER.xlsx \
    ../ngoma_charts_frontend/ngoma_deploy/src/data/chartData.js

# Skip the merge pass (master workbook only, no canonicalization)
python scripts/export_frontend_data.py charts/seed_data/Ngoma_Charts_MASTER.xlsx \
    ../ngoma_charts_frontend/ngoma_deploy/src/data/chartData.js --no-merge

# Explicit merge workbook path
python scripts/export_frontend_data.py charts/seed_data/Ngoma_Charts_MASTER.xlsx \
    ../ngoma_charts_frontend/ngoma_deploy/src/data/chartData.js \
    --merge-workbook Data/ngoma_duplicate_releases_final_merge_ready.xlsx
```

## Architecture

### Two independent data paths

The project has two completely separate ways data enters the system — they share models but have different entry points:

**Path 1 — Master workbook (`seed_data` / `master_dataset.py`):**  
`Ngoma_Charts_MASTER.xlsx` → `charts/master_dataset.py` → Django ORM (`Release`, `MonthlyChartEntry`, etc.)  
This is the canonical historical data import. `master_dataset.py` does all deduplication and canonicalization before touching the DB. The `seed_data` management command calls `import_master_workbook()`, which clears and rebuilds everything from scratch.

**Path 2 — Weekly uploads (`pipeline.py`):**  
Admin uploads a weekly `.xlsx` → `process_weekly_upload()` in `charts/pipeline.py` → `PlatformChartEntry` → `rebuild_monthly_chart()` → `MonthlyChartEntry`  
This is the live-operations path. Normalization rules come from `NormalizationRule` DB records (plus hardcoded fallbacks in `pipeline.py`).

These two paths are **not automatically kept in sync**. The master workbook is the source of truth for the seeded/historical data; weekly uploads layer on top of it.

### Release deduplication key

`master_dataset.py` uses `_release_key(title, primary_artist)` → `(normalized_title, artist_key)` as the identity key throughout canonicalization. Two rows with different raw spellings but the same key are merged into one release. Any duplicate-merge work must operate at this key level before data enters the DB, because `Release` has `unique_together = ['canonical_title', 'artist', 'chart_type']`.

`pipeline.py` uses a similar but simpler key via `normalize_entry()`, stored as `canonical_title = title.lower().strip()` on the `Release` model.

### Public data delivery

The public frontend can be served in two modes:

1. **Static file** (`chartData.js`): Generated by `scripts/export_frontend_data.py` reading the master workbook and then applying the duplicate-release merge workbook (`Data/ngoma_duplicate_releases_final_merge_ready.xlsx`) to canonicalize titles and artist credits. The frontend repo has this file at `src/data/chartData.js`. It is committed and does not auto-update when the DB changes.

2. **Live API** (`/api/v1/app-data/`): `charts/app_data.py` builds a single JSON payload containing the full chart data, artists, releases, certifications, news, settings, and page content from the DB. This is the production API path. Includes a `revision` field derived from `AuditLog` that the frontend uses for cache-busting.

### Key files

| File | Purpose |
|---|---|
| `charts/models.py` | All Django models. `Release.canonical_title` is the dedup key. `MonthlyChartEntry.platform=None` means combined chart. |
| `charts/master_dataset.py` | Master workbook ingestion: parses, canonicalizes, and imports the five-sheet workbook. Also used by `export_frontend_data.py`. |
| `charts/pipeline.py` | Weekly upload processing and monthly aggregate rebuilds. |
| `charts/app_data.py` | `/api/v1/app-data/` view — builds the full public JSON payload from the DB. |
| `charts/artist_credits.py` | `release_credit_payload()` — the single function that formats primary/featured artist display strings used everywhere in the API. |
| `charts/cms_serializers.py` | DRF serializers for the CMS. `CmsReleaseSerializer` handles `primary_artist_ids`/`featured_artist_ids` as write-only inputs synced to `ReleaseArtistCredit`. |
| `charts/cms_views.py` | CMS API viewsets. Every mutating action calls `audit()` from `cms_utils.py` — this is what increments the `revision`. |
| `charts/cms_utils.py` | `audit()`, `publish_chart_upload()`, `recalculate_certifications()`. |
| `charts/artist_metadata.py` | Static `artist_country()` lookup for auto-populating country/country_code on import. |
| `scripts/export_frontend_data.py` | Standalone script (no Django): reads the master workbook and writes `chartData.js`. |
| `ngoma_backend/settings.py` | SQLite locally, PostgreSQL via `DATABASE_URL`. Cloudinary for media when `CLOUDINARY_URL` is set. |

### Master workbook schema

The master workbook must contain exactly five sheets: `README`, `Singles_Combined`, `Albums_Combined`, `Singles_Platforms`, `Albums_Platforms`. `load_master_workbook()` validates headers and enforces that each combined sheet has exactly 50 ranks per month for all 9 months (900 combined rows total).

### Certification thresholds

Two threshold sets exist — be careful not to confuse them:
- `master_dataset.py` `CERTIFICATION_THRESHOLDS`: `{gold: 200, platinum: 400, diamond: 600}` — used during master import (Display_Points-based, max 50 per month).
- `models.py` `Certification.THRESHOLDS`: `{gold: 5000, platinum: 10000, diamond: 20000}` — used by `pipeline.py`'s `award_certifications()` for weekly-upload path.

### Database

SQLite locally (`ngoma_charts.db`, gitignored). PostgreSQL in production via `DATABASE_URL`. The `.env` file (also gitignored) is used for local overrides; `.env.example` documents the available variables.

### CMS authentication

Session-based (not token-based). The CMS frontend fetches a CSRF token from `/api/v1/cms/csrf/` on load. All mutating CMS requests must include `X-CSRFToken` and `credentials: "include"`.

### Management commands location

All custom management commands live in `charts/management/commands/`. New commands go there.
